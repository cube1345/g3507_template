下面是推荐实施方案。该改动会影响多文件和核心逻辑，我先明确受影响范围、潜在副作用与落地策略。

## 1. 受影响文件与潜在副作用

### 受影响模块
需要把以下旧底层驱动改成和当前 [`led`](MyUsed/MyDevices/led_driver.h:1)、[`key`](MyUsed/MyDevices/key_driver.h:1)、[`encoder`](MyUsed/MyDevices/encoder_driver.h:1)、[`buzzer`](MyUsed/MyDevices/buzzer_driver.h:1) 一致的分层：

- OLED：当前集中在 [`MyUsed/MyDrivers/OLED_driver.h`](MyUsed/MyDrivers/OLED_driver.h:1)、[`MyUsed/MyDrivers/OLED_driver.c`](MyUsed/MyDrivers/OLED_driver.c:1)、[`MyUsed/MyInts/OLED_int.c`](MyUsed/MyInts/OLED_int.c:1)
- ICM42688：当前集中在 [`MyUsed/MyDrivers/icm42688_driver.h`](MyUsed/MyDrivers/icm42688_driver.h:1)、[`MyUsed/MyDrivers/icm42688_driver.c`](MyUsed/MyDrivers/icm42688_driver.c:1)、[`MyUsed/MyInts/icm42688_int.c`](MyUsed/MyInts/icm42688_int.c:1)
- Xunji：当前集中在 [`MyUsed/MyDrivers/xunji_driver.h`](MyUsed/MyDrivers/xunji_driver.h:1)、[`MyUsed/MyDrivers/xunji_driver.c`](MyUsed/MyDrivers/xunji_driver.c:1)、[`MyUsed/MyInts/xunji_int.c`](MyUsed/MyInts/xunji_int.c:1)
- Motor：当前集中在 [`MyUsed/MyDrivers/motor_driver.h`](MyUsed/MyDrivers/motor_driver.h:1)、[`MyUsed/MyDrivers/motor_driver.c`](MyUsed/MyDrivers/motor_driver.c:1)
- Stepper：当前集中在 [`MyUsed/MyDrivers/stepper_driver.h`](MyUsed/MyDrivers/stepper_driver.h:1)、[`MyUsed/MyDrivers/stepper_driver.c`](MyUsed/MyDrivers/stepper_driver.c:1)

### 潜在副作用
- 这些模块现有业务层可能直接 include 旧头文件，例如 [`MyUsed/MyLibs/OLED_menu.c`](MyUsed/MyLibs/OLED_menu.c:2) 使用 [`OLED_int.h`](MyUsed/MyInts/OLED_int.h:1)，[`MyUsed/MyLibs/IMU.c`](MyUsed/MyLibs/IMU.c:4) 使用 [`icm42688_driver.h`](MyUsed/MyDrivers/icm42688_driver.h:1)。
- 为降低风险，旧路径 [`MyUsed/MyDrivers/*.h`](MyUsed/MyDrivers:1) 和 [`MyUsed/MyInts/*.h`](MyUsed/MyInts:1) 不直接删除，而是改成兼容壳，转发到新的 Facade 或 Board 层。
- OLED 现有 [`OLED_driver.c`](MyUsed/MyDrivers/OLED_driver.c:1) 文件很大，直接一次性迁移全部绘图函数风险较高。更稳妥做法是先把“对象化初始化、I2C 线控制、板级绑定、兼容入口”拆出来，保留原显示 API 名称，减少业务层破坏。

## 2. ops 继承设计

不同外设语义不同，不强行套同一个 ops：

1. OLED 是显示设备，新增：
   - `device_display_ops_t`
   - 操作：`init`、`update`、`clear`
   - OLED 自身 typedef 为 `oled_base_ops_t`

2. ICM42688 是 IMU/传感器设备，新增：
   - `device_sensor_ops_t`
   - 操作：`init`、`read_raw`、`read_scaled`
   - ICM42688 自身 typedef 为 `icm42688_base_ops_t`

3. Xunji 是多路数字输入阵列，复用或扩展输入语义更合适：
   - 新增 `device_array_input_ops_t`
   - 操作：`read_all`
   - Xunji 自身 typedef 为 `xunji_base_ops_t`

4. Motor 是执行器，类似速度/功率输出设备：
   - 新增 `device_motor_ops_t`
   - 操作：`start`、`stop`、`set_pwm`、`velocity_calc`

5. Stepper 是带方向、频率、启停状态的运动执行器：
   - 新增 `device_stepper_ops_t`
   - 操作：`enable`、`set_direction`、`set_frequency`、`start`、`stop`、`service`

## 3. 推荐文件结构

### Core 层
修改：
- [`MyUsed/MyCore/device_ops.h`](MyUsed/MyCore/device_ops.h:1)

新增上述 display/sensor/array_input/motor/stepper ops。

### Device 层
新增或迁移：
- `MyUsed/MyDevices/oled_driver.h/.c`
- `MyUsed/MyDevices/icm42688_driver.h/.c`
- `MyUsed/MyDevices/xunji_driver.h/.c`
- `MyUsed/MyDevices/motor_driver.h/.c`
- `MyUsed/MyDevices/stepper_driver.h/.c`

职责：
- 定义对象结构体：例如 `oled_t`、`icm42688_t`、`xunji_t`、`motor_t`、`stepper_t`
- 定义 cfg：硬件回调、GPIO、定时器实例、量程配置等
- 实现对象方法和 ops 分发
- 尽量用 `static inline` 做小的参数检查、状态转换、饱和限幅等

### Board 层
新增：
- `MyUsed/MyBoards/oled_board.h/.c`
- `MyUsed/MyBoards/icm42688_board.h/.c`
- `MyUsed/MyBoards/xunji_board.h/.c`
- `MyUsed/MyBoards/motor_board.h/.c`
- `MyUsed/MyBoards/stepper_board.h/.c`

职责：
- 实例化板级对象
- 绑定 SysConfig 宏、GPIO、Timer、I2C、PWM 等强板级资源
- 将原来 [`*_int.c`](MyUsed/MyInts:1) 中的板级绑定逻辑迁移到 board 层
- 中断函数或强板级 ISR 也放到 board 层，例如 stepper 的 `STEP_CTRL_TIMER_INST_IRQHandler`

### Facade 层
新增：
- `MyUsed/MyFacades/oled_facade.h/.c`
- `MyUsed/MyFacades/icm42688_facade.h/.c`
- `MyUsed/MyFacades/xunji_facade.h/.c`
- `MyUsed/MyFacades/motor_facade.h/.c`
- `MyUsed/MyFacades/stepper_facade.h/.c`

职责：
- 保持原业务 API 名称不变，例如：
  - `OLED_Init`、`OLED_Update`、`OLED_Clear`、`OLED_ShowString` 等
  - `icm42688_init`、`bsp_Icm42688Init` 等
  - `Xunji_Init`、`Xunji_Get`
  - `Motor_Init`、`Set_PWM`、`Velocity_A/B`
  - `Stepper_Init`、`Stepper_Start/Stop/...`
- 内部转调 board 对象和 device 方法。

### Compat 层
修改旧文件为兼容壳：
- [`MyUsed/MyDrivers/OLED_driver.h`](MyUsed/MyDrivers/OLED_driver.h:1) -> include `../MyFacades/oled_facade.h`
- [`MyUsed/MyDrivers/icm42688_driver.h`](MyUsed/MyDrivers/icm42688_driver.h:1) -> include `../MyFacades/icm42688_facade.h`
- [`MyUsed/MyDrivers/xunji_driver.h`](MyUsed/MyDrivers/xunji_driver.h:1) -> include `../MyFacades/xunji_facade.h`
- [`MyUsed/MyDrivers/motor_driver.h`](MyUsed/MyDrivers/motor_driver.h:1) -> include `../MyFacades/motor_facade.h`
- [`MyUsed/MyDrivers/stepper_driver.h`](MyUsed/MyDrivers/stepper_driver.h:1) -> include `../MyFacades/stepper_facade.h`

对应旧 `.c` 文件可以：
- 保留为空兼容翻译单元，只 include 自己头文件，避免重复定义。
- 真正实现迁移到 `MyDevices` / `MyBoards` / `MyFacades`。

## 4. 实施顺序

1. 修改 [`device_ops.h`](MyUsed/MyCore/device_ops.h:1)，新增各类 ops。
2. 先做较小模块：Xunji、Motor。
3. 再做 Stepper：把对象状态从旧全局数组迁移到 `stepper_t` 对象，ISR 放 board 层，facade 保持 API。
4. 再做 ICM42688：保留现有算法和寄存器访问函数语义，但对象化 `base/ops/cfg/current_config`。
5. 最后做 OLED：保留绘图 API，对底层 I2C 线控制和初始化入口做对象/board/facade 拆分。
6. 修改旧 Drivers/Ints 文件为兼容壳。
7. 编译/语法检查，优先修复：头文件同名包含、重复符号、旧 API 找不到、ISR 重复定义。

## 5. 建议的第一轮落地范围

为了降低一次性重构风险，第一轮我会做到：
- 完成五个模块的 `MyDevices`、`MyBoards`、`MyFacades` 文件骨架和主要对象封装。
- 保持所有原业务 API 名称可用。
- 对 OLED/ICM42688 这类大文件，优先保证兼容性和分层边界，不强行重写全部显示算法/传感器算法。
- 每个新增/修改的公开接口补充 Doxygen 注释。
